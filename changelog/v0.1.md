## 1. О чём этот проект простыми словами

**alpha-trader** — это не «бот по свечным паттернам» и не подгонка индикаторов под прошлый год. Идея другая:

На Polymarket есть короткие бинарные рынки вроде **«BTC выше/ниже через 5 минут»**. Цена YES-токена — это рыночная оценка вероятности «Up». На Binance в это же время крутится **реальный спот BTC** (и другие монеты).

**Гипотеза:** когда спот на Binance уже сдвинулся, а стакан Polymarket **ещё не догнал**, между «справедливой» вероятностью и ценой PM возникает **временный разрыв (gap)**. Если разрыв живёт дольше, чем ваша задержка до ордера, теоретически можно покупать «дешёвый» YES (или продавать дорогой) до того, как PM схлопнет lag.

Чтобы не гадать, проект строится **слоями**:

1. Сначала — **математика**: \(p^\*\) из спота, времени до конца окна \(\tau\) и волатильности \(\sigma\) (модель GBM, без z-score и momentum для fair).
2. Потом — **проверка на истории Binance**: калибруется ли \(p^\*\) хоть как-то разумно.
3. Потом — **неделя наблюдений без ордеров** (market logger): реальный lag PM, размер gap, go/no-go.
4. Только после этого — симулятор, бэктест, лайв-бот (фазы 2–6, **ещё не сделаны**).

Сейчас (v0.1) готовы **фазы 0, 1, 1b** и **выкладка на VPS**. Торговли нет.

## 2. Идея и экономический смысл

### 2.1 Что такое \(p^\*\)

**\(p^\*\)** — ответ на вопрос: *«При текущей цене спота и оставшемся времени до конца 5-минутного окна, какой шанс, что в конце окна цена будет выше \(S_0\)?»*

Входы:

| Величина | Откуда |
|----------|--------|
| \(S_t\) | Mid спота с Binance (bookTicker) |
| \(S_0\) | Спот на **открытии окна** (в v0 — упрощённо; в phase 3 — по календарю PM) |
| \(\tau\) | Секунд до конца окна |
| \(\sigma\) | Годовая волатильность (EWMA по недавним ценам) |

Формула — классический **GBM** (через \(d_2\) и нормальную CDF). Есть также **Monte Carlo** для сверки. Это **механистическая** fair price, не «после такой свечи в 62% случаев был рост».

### 2.2 Что такое \(p_{\text{mkt}}\) и gap

**\(p_{\text{mkt}}\)** — mid (или microprice) токена YES на Polymarket.

**gap_level** = \(p_{\text{mkt}} - p^\*\).

- Если PM **ниже** \(p^\*\) — рынок «отстаёт вверх» → гипотетически покупка YES.
- Если **выше** — продажа YES / покупка NO.

Рукав **A (арбитраж лага)** как раз про это. Рукава B (MM) и C (rebates) в v0.1 **не реализованы** — только заглушки в коде.

### 2.3 Чем мы сознательно НЕ занимаемся

- Паттерны OHLCV, z-score, momentum для **fair**.
- Grid search сотен параметров под PnL одного года.
- Смешивание цены BTC и вероятности YES в «одном стакане».
- Торговля до измерения реального lag на живом PM.

Всё это прописано в `STRATEGY.md` §2 и §12–§16.

### 4.2 Фаза 0 — ядро fair

- **`fair.py`** — `p_up_gbm`, `p_up_mc`.
- **`vol.py`** — `sigma_ann_from_closes` (EWMA по log-return).
- **`tests/test_fair.py`** — границы: ATM ≈ 0.5, ITM/OTM, экспирация, MC ≈ GBM.

**Проверка:** `pytest tests/test_fair.py` — проходит.

### 4.3 Фаза 1 — история Binance без PM

- **`clock.py`** — окна 5m, `tau_sec`, загрузка окон из JSON (для phase 3).
- **`spot.py`** — чтение klines JSON.
- **`symbols.py`** — список из 7 пар по умолчанию, разбор `BINANCE_SYMBOLS`, маппинг PM slug/token по символам.
- **`scripts/fetch_binance_klines.py`** — скачивание с Binance Vision (ZIP → JSON по годам), флаг `--symbols`.
- **`tools/run_calibration.py`** — Brier score и reliability bins **по каждому символу**.

**Упрощение v0:** одно 5m-окно = одна свеча Binance; \(S_0\) = open; для предсказания без look-ahead: \(S_t\) = close **предыдущей** свечи, \(\tau\) = 300 с. Это честно, но **слабый** сигнал на таком разрешении.

**Локально скачано:** `data/binance/btcusdt_5m_2024.json` (~105k баров). Остальные символы — командой fetch при необходимости.

### 4.4 Фаза 1b — наблюдение рынка (без ордеров)

- **`feeds/binance_ws.py`** — одиночный и **комбинированный** WS (`BinanceMultiBookTickerFeed`) на все символы одним подключением.
- **`feeds/gamma.py`** — slug Polymarket → token id YES/NO.
- **`feeds/pm_clob.py`** — стакан PM (REST + WS), mid/microprice.
- **`market_logger_service.py`** — основной цикл: для **каждого символа** пишет строку JSONL с полем `"symbol"`.
- **`tools/market_logger.py`** — CLI, `.env`, `--mock-pm` / live.
- **`tools/analyze_market_logs.py`** — отчёт **сводный + по символам**, `go_arb`, `lag_pm_ms`, `config_recommendation.json`.
- **`health.py`** — `output/health/logger.json` для мониторинга на VPS.

**Режимы логгера:**

| Режим | Когда | Что делает |
|-------|-------|------------|
| **`--mock-pm`** | CI, локальный smoke | Синтетические Binance + PM, **без сети** |
| **Live** | VPS, production | Реальный WS Binance; PM — по `PM_MARKET_SLUGS` / token id; без slug — mock PM только для этого символа |

### 4.5 Семь символов Binance (v0.1)

По умолчанию (если не задан `BINANCE_SYMBOLS`):

`BTCUSDT`, `ETHUSDT`, `SOLUSDT`, `XRPUSDT`, `DOGEUSDT`, `HYPEUSDT`, `BNBUSDT`

Для каждого в логе своя строка с `symbol`, своим \(p^\*\), gap (если PM настроен или mock). На VPS один процесс, один combined WebSocket Binance — экономия соединений.

Polymarket настраивается **по символу**:

```bash
PM_MARKET_SLUGS={"BTCUSDT":"ваш-slug-btc-5m"}
# или PM_MARKET_SLUG_ETHUSDT=... / PM_YES_TOKEN_ID_BTCUSDT=...
```

### 5.2 Smoke логгера (mock PM)

Все прогоны ниже — **синтетика**. PM **намеренно** отстаёт от \(p^\*\). **`go_arb: true` здесь не доказывает прибыль.**

| Прогон | Длительность | Файл лога | Отчёт |
|--------|--------------|-----------|-------|
| Smoke BTC | 60 с | `output/logs/smoke.jsonl` | `output/reports/smoke/` |
| Deploy smoke | ~5 с | `output/logs/deploy_smoke.jsonl` | `output/reports/deploy_smoke/` |
| Multi 7 sym | 3 с | `output/logs/multi_smoke.jsonl` | `output/reports/multi_smoke/` |

**Пример smoke BTC (60 с, mock):**

- Строк в логе: 232  
- Средний \|gap\|: ~0,015  
- Доля \|gap\| > 2%: ~27%  
- lag p50 / p95 (эвристика): ~281 / 425 ms  
- **go_arb: true** — только проверка, что анализатор и пороги не сломаны  

**Multi-symbol mock:** 84 строки (12 на символ × 7), в отчёте таблица по каждой паре — проверка, что мульти-логгер пишет и режет отчёт корректно.

В отчётах: `summary.md` + `config_recommendation.json` (`lag_pm_ms_p50/p95`, `go_arb`, `min_edge_suggested` = 0.03 по умолчанию).

### 5.3 Есть ли «зацеп» за прибыльность?

**Честный ответ на v0.1: нет подтверждённого edge.**

| Что проверено | Результат |
|---------------|-----------|
| Математика \(p^\*\) | OK (тесты) |
| Калибровка на истории | Brier = случайное угадывание |
| Реальный lag PM | **Не измеряли** (не было live-недели) |
| PnL после комиссий | **Не считали** (фаза 2 не начата) |

**За что можно зацепиться концептуально (гипотеза, не факт):**

- Логика «спот двигается → вероятность должна сдвинуться → PM иногда догоняет с задержкой» — разумна для коротких бинарок.
- Проект **правильно** ставит измерение **до** торговли (1b), а не сразу бота.
- Инфраструктура готова: один VPS, мульти-символ, health, logrotate.

**Что должно появиться, чтобы говорить о деньгах:**

1. **Live 1b, 7–10 дней:** \|gap\| бывает > 2–3%, живёт **дольше** вашей p95-латентности; \(p^\*\) на settle-строках Brier < 0.25 и лучше 0.5.  
2. **Фаза 2:** сим с `lag_pm_ms` из отчёта 1b, fee, slippage — edge ≥ 0 не «на глаз».  
3. **Фаза 3:** правильные \(S_0\), \(\tau\) по правилам PM, не «свеча Binance ≈ окно PM».
